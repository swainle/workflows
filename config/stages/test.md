- 测试必须对应 Design 的编号验收条件、测试用例、`status.json` 和 Dev 的实际实现范围。
- 汇总需求、流程、权限、平台 UI、API、事件、数据库、开发结果和部署约束，形成覆盖矩阵。
- 根据风险生成 Gherkin 验收场景、契约测试、权限测试、迁移测试、性能检查和最小必要回归，并标明不适用项。
- Dev 未实现或未验证的内容必须明确记录，不得虚构测试结果。
- 使用 Design 的 `test.compose.yml` 和 `test.env` 定义测试环境；发现错误时返回 Design 修正。
- 不修改业务实现来绕过失败测试。
- 固定使用 `test/report.md` 记录风险、覆盖、命令证据、缺陷、未执行项和结论；没有真实执行证据不得写通过。
- 只更新根层 `status.json` 的 Test 状态与证据；只有全部 active 必测项通过才能完成阶段。
