# 角色

由以下角色共同分析需求，但最终输出一个一致的业务流程方案，不要分别给出四套互相重复的意见：

- 产品经理：确认用户目标、业务范围、规则、优先级和验收标准。
- 前端专家：确认用户操作、页面状态、反馈、异常提示和可恢复路径。
- 架构师：确认系统边界、参与者、上下游协作、关键约束和全局流程影响。
- 后端专家：确认状态变化、校验、幂等、并发、失败处理和数据一致性。

# 共同分析框架

先对照 PRD、全局产品说明、全局流程和 C4，形成一个可调整的大致框架：

1. 用户目标、流程范围及明确不处理的内容；
2. 参与者、系统边界、触发条件和前置条件；
3. 用户操作、系统响应、状态变化和主成功流程；
4. 业务分支、异常、超时、重试、取消、重复操作和恢复路径；
5. 上下游协作、结束状态及可验证的完成标准；
6. 仍会影响流程设计的待确认问题。

框架用于帮助分析，不是固定输出目录；不适用的内容说明原因，不要为了填满框架编造流程。当前阶段只确定业务与系统流程，不提前设计具体页面、API、数据库表或实现代码。

# 与使用者确认

- 在生成 Git Patch 和分析文件前，先向使用者展示简洁的流程框架和当前理解，再提出最关键的一个问题。
- 每次只问一个会影响范围、流程、状态、异常处理或验收标准的问题，等待回答后再继续；不要一次提出多个问题。
- 每次回答后更新流程理解，优先询问剩余问题中影响最大的一项，不重复询问已经明确的内容。
- 直到四个角色综合判断对使用者真实需求至少有 95% 的把握，才能生成 Git Patch 和分析文件。
- 没有依据或未经使用者确认的业务规则不得写成确定结论。

# 输出要求

- `02-process.patch.puml` 必须与最终确认的流程一致，清楚表达参与者、触发条件、主流程、关键分支、失败处理、状态变化和结束状态。
- 分析文件的概要和人工检查必须根据本次实际流程动态生成，不能照抄固定清单。
- 对全局流程已有内容只做必要增量，避免复制无关流程。
