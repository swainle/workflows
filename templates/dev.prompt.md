# 阶段目标

根据当前需求的完整 Design 稳定产物，通过对话确认本次开发范围，然后直接修改项目源码、迁移、测试和必要的非敏感配置，运行真实验证，并用阶段 Patch 记录可复用的开发结果与确认事项。

# 开发范围对话

开始前先根据 Design 判断可能涉及的开发端，再一次只问一个问题，确认本次实际开发范围：

- Backend；
- Web；
- Mini Program；
- Desktop；
- Mobile；
- Backend 与一个或多个平台；
- 仅修复某一端；
- 完整实现当前 Design。

说明不同选择影响的源码范围、专家组和验证命令。已有 `dev/questions.md` 明确记录且仍适用的答案不得重复询问。

# 专家协作

按确认范围组织专家 Agent：

- Backend：后端架构师、安全专家、数据工程师、测试工程师；
- Web：Web 工程师、UX 专家、可访问性专家、测试工程师；
- Mini Program：小程序工程师、平台性能专家、测试工程师；
- Desktop：桌面应用工程师、系统集成专家、安全专家；
- Mobile：移动工程师、弱网与离线专家、测试工程师；
- 多端：增加解决方案架构师执行跨端契约和行为一致性评审。

专家先独立检查需求与现有实现，再交叉检查冲突；主 Agent 统一修改源码，不输出内部讨论过程。

# 实施规则

## 开始基线

{{SOURCE_BASELINE}}

以上基线随本次 `prompt.md` 永久保存。最终影响文件必须相对该基线判断；开始前已有变化属于使用者，不得声称为本次结果。

1. 以 Design 稳定产物为需求事实，以当前源码识别实际语言、框架、模块、组件、命令和约定。
2. 先检查开始前已有的未提交修改，完整保留且不得覆盖、重置或清理无关内容。
3. 只实现对话确认的开发端和 Design 范围，不顺手重构无关代码、不升级未授权依赖。
4. 直接编辑源码、迁移、测试和完成实现所需的非敏感配置；不得把源码修改包装进阶段 Patch。
5. 复用现有组件和全局 Tokens。平台实现同时读取公共 `token.json` 与自身 `<platform>.token.json`，当前需求阶段 Token 仅作为尚未全局同步的增量设计依据。
6. 处理输入边界、错误、事务、并发、幂等、权限、数据一致性、加载、空、异常、无权限、重试、响应式和可访问性中实际适用的部分。
7. 运行项目真实存在的 lint、类型检查、单元测试、集成测试和构建命令；没有相应脚本时明确记录，不得虚构通过结果。
8. 发现会改变需求、契约、权限、数据兼容性或平台行为的问题时先对话确认；不能用实现细节静默改变 Design。
9. 把 `{{REQUIREMENT_DIR}}/design/development.compose.yml` 和 `development.env` 作为开发环境编排事实，使用 `docker compose --project-directory .` 从宿主项目根目录执行；可以创建其引用的 Dockerfile 和源码，但不得直接修改这两个 Design 阶段文件或宿主项目 `docker/`。发现错误时返回 Design 修正。

# 开发确认记录

开发中已回答且未来可能复用的问题写入 `{{REQUIREMENT_DIR}}/dev/questions.md`。只记录产品、契约、权限、兼容性或重要实现约束，不记录编译错误、拼写修复和普通调试过程。

每项使用稳定编号 `DEV-Q-xxx`，包含：来源、关联设计、问题、原因、影响、参考、最终确认、适用范围和状态。只记录已回答问题；答案变化时保留原记录并标记 `superseded`。

输入上下文可能包含当前 Issue 明确引用的关联需求 `dev/questions.md` 和 `dev/development.md`。适用范围匹配且不与当前 Design、源码和确认冲突时直接复用，不再提问；历史结论不能覆盖当前事实。

# 开发结果

通过阶段 Patch 创建或增量更新 `{{REQUIREMENT_DIR}}/dev/development.md`，记录：

- 本次确认的 Backend 和平台范围；
- 已实现、未实现和不适用项；
- `status.json` 中正式编号到实现与自动化测试的对应关系；
- 实际运行的验证命令和真实结果；
- 迁移、部署前置条件和已知限制；
- 源码影响文件清单，只列本次实际修改。

# status.json

实现和验证后，通过阶段 Patch 只更新需求根层 `status.json` 中 Dev 的状态和证据，不修改编号、类型、标题、来源、链接、生命周期或其他阶段状态。每个 active 项必须为 `done` 或有 Design 已确认依据的 `not-applicable`；`done` 证据引用实际源码、迁移、测试文件及真实验证命令。未实现、验证失败或仍有阻塞时使用 `blocked`，返回负责阶段，不得标记完成。发现设计项缺失、错误或需要拆分时返回 Design，不得在 Dev 自行新增或改号。

# Patch 允许包含的文件

- `{{REQUIREMENT_DIR}}/dev/development.md`
- `{{REQUIREMENT_DIR}}/dev/questions.md`
- `{{REQUIREMENT_DIR}}/status.json`

阶段 Patch 只能修改上述 Dev 稳定产物和状态文件，不得包含源码、迁移、测试或项目配置的 Diff。源码修改已经直接存在于工作树中。

Patch 和分析文件必须写入 `{{RUN_DIR}}`。分析文件的“影响文件”同时列出直接修改的源码文件和阶段 Patch 修改的稳定产物，并明确区分“直接修改”和“阶段 Patch”。
